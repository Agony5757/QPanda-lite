## 1) `CircuitControlContext` 仅写 `circuit_str`，不进 `opcode_list`（未修复）

### 症状

任何使用

```python
with circuit.control(*controls):
    circuit.some_gate(target)
```

的代码路径，在通过 `circuit.qasm` / `circuit.originir` 导出电路时 **CONTROL 块
整段丢失**——块内的门被当作**无条件**操作序列化。对模拟器而言相当于静默退化
成不受控版本。

### 根因（已定位）

- `qpandalite/circuit_builder/qcircuit.py:42-47` 的 `CircuitControlContext.__enter__/__exit__`
  只把 `CONTROL q[..]` / `ENDCONTROL` 文本追加到 `self.c.circuit_str`。
- `qpandalite/circuit_builder/qcircuit.py:121-125` 的 `_make_qasm_circuit`
  （以及 `_make_originir_circuit`）只从 `self.opcode_list` 翻译，**完全不读
  `circuit_str`**。
- `circuit.z(target)` 等门走 `add_gate`，写入 `opcode_list` 的是**裸 gate**
  （`control_qubits=None`），上下文管理器无法追溯修改。

### 影响范围

- Grover（`grover_oracle.py`）：本次 PR #139 commit `00fe5eb` 内已**绕开**
  （改用 `_apply_mcx` 直接通过 `add_gate(..., control_qubits=[...])` 路径表达
  受控语义）。
- 其他调用点待全仓审计：
  ```
  grep -rn 'with .*\.control(' qpandalite/
  ```
  凡是结果在 `qpandalite/algorithmics/` 下的都有同类隐患。

### 建议修复

**上游级修复**（推荐）：修改 `CircuitControlContext.__exit__` 或 `Circuit.control`，
让块内添加的每条 opcode 自动继承 `control_qubits`。具体思路：

- `__enter__` 记录进入前的 `len(self.c.opcode_list)`。
- `__exit__` 回填区间内 opcode 的 `control_qubits` 字段。

这样 `with circuit.control(a, b): circuit.z(c)` 自然会产出一条 opcode
`("Z", c, None, None, False, [a, b])`，QASM/OriginIR 导出天然正确。

此修复属于 circuit_builder 核心改动，**不在本 PR scope**，建议独立 issue + PR
处理。

## 2) `_apply_mcx` 只支持 ≤3 个 control（新引入约束）

### 来源

`00fe5eb` 在 `qpandalite/algorithmics/circuits/grover_oracle.py` 新增的
`_apply_mcx` helper，目前只有 0/1/2/3 control 的分支，4+ 控会直接抛
`NotImplementedError`。

### 影响

- 当前测试只用 n=3 data qubits 的 Grover（oracle 命中 3-control，diffusion
  命中 2-control），全部覆盖。
- 未来若有 **4+ qubit Grover 集成测试** 或用户在自己代码里用 `grover_oracle`
  / `grover_diffusion` 传 ≥5 个 data qubits，会立即抛错。
- 相比之前 B 版本"静默空转"属于**进步**（显式失败好于静默错误），但仍是能力
  缺口。

### 建议修复

实现通用 MCX ladder，选一个方案：

1. **Gidney 的 temporary-AND tree**（`arXiv:1709.06648`）：用 `floor(n/2)-1`
   个临时 ancilla，深度 `O(log n)`，电路计数最小。
2. **V-chain with borrowed ancilla**（Nielsen & Chuang Fig 4.10）：需要
   `n-2` 个借用 ancilla（可脏），深度 `O(n)`。

任一方案都可以扩容 `_apply_mcx`，独立 PR 处理。

## 3) `grover_diffusion.ancilla` 参数退化为死参数

### 现状

`00fe5eb` 把 `_apply_mcz` 的 target 从 ancilla 改成 `qubits[-1]`（MCZ 所有位
对称，语义等价）。签名上 `ancilla` 仍保留为可选参数，但函数体内**不再使用**。

文档（docstring）已标注"Not needed when *qubits* has length 1"，但未注明
"len(qubits) >= 2 时也同样不使用"。

### 建议处理

- 短期：更新 docstring 明确"当前实现未使用 ancilla 参数；保留仅为签名向后兼
  容"。
- 长期：下一次允许 API break 的 PR 里去掉该参数，或改用 `warnings.warn(...,
  DeprecationWarning)` 引导用户。

不影响正确性，**优先级低**。

## 4) `dicke_state` 当前用 SBM，丧失 SCUC 线性深度（性能欠账）

### 现状

`qpandalite/algorithmics/circuits/dicke_state.py` 在 k≥2 时走 `rotation_prepare`
（SBM 分解，线路深度指数级），原 SCUC cascade 只能处理 k=1。

### 根因

SCUC 需要 multi-controlled Ry（MCRy），`circuit_builder` 现未原生支持该门。
多次尝试在现有 API 上修 SCUC 后放弃，改为 SBM 兜底（见 PR #139）。

### 建议修复

1. 在 `circuit_builder` 加 MCRy gate 支持（横穿 spec → parser → opcode → simulator）。
2. 按 Bärtschi & Eidenbenz 2019 重写 `_dicke_unitary` 以使用真正的 SCUC cascade，
   达到 O(n) 线路深度。
3. 独立 PR，不在当前 #139 scope。

**优先级低**（正确性已保证，仅深度不优；未有用户报实际性能问题）。

## 5) `shadow_expectation` 对每 snapshot 做 O(2^n) 遍历（性能欠账）

### 现状

`qpandalite/algorithmics/measurement/classical_shadow.py:286-311`：
当前实现把 snapshot 从"采样 1 outcome"改为"存整个 counts 字典"（PR #139 修 variance 时引入）。
正确性已验证，但每个 snapshot 的 aligned 期望值计算会遍历全部 `counts` 条目（O(2^n) 次乘加）。

### 影响

- n ≤ 5 时无感知（100 snapshot × 32 entries ≈ 3200 次运算）。
- n ≥ 6（64+ entries / snapshot）且 `n_shadow` 较大时，计算成本显著上升。

### 建议修复

对同一 `(pauli_string, basis)` 组合在同一 snapshot 内的贡献做 per-basis 缓存，
避免对不同 Pauli string 重复遍历同一 `counts`。独立 PR，**触发条件**：出现
n ≥ 6 的测试用例或用户报 shadow 估计慢。

**优先级低**。
